# Product Requirements Document: Undergraduate Student Study Survey

## Overview

Build a short web-based survey targeted at undergraduate business students that collects information about their study habits and productivity. The app has two pages: a survey form and a results page showing aggregated responses. Responses are stored in a Supabase PostgreSQL database.

---

## Tech Stack

* **Framework:** Vite + React
* **Database:** Supabase (PostgreSQL)
* **Hosting:** Azure Static Web Apps
* **Charting:** Recharts

---

## Survey Questions

The survey must include the following four questions, using the specified input types:

| # | Question                                 | Input Type                                          |
| - | ---------------------------------------- | --------------------------------------------------- |
| 1 | What is your favorite place to study?    | Text input box                                      |
| 2 | What is your grade level?                | Dropdown list (freshman, sophomore, junior, senior) |
| 3 | How many hours do you study per day?     | Radio buttons                                       |
| 4 | What study methods do you use regularly? | Checkboxes + conditional text input                 |

### Question Details

**Q1 — Favorite place to study**

* Free-text input
* Placeholder: "e.g. Library"
* Autofocus
* Required field

**Q2 — Grade level**

* Dropdown list containing all college grade levels (Freshman, Sophomore, Junior, Senior)
* Required field

**Q3 — Study Hours**

* Radio button group with the following options (select one):

  * Less than 1 hour
  * 1–2 hours
  * 2–3 hours
  * More than 3 hours
* Required field

**Q4 — Study Methods (select all that apply)**

* Checkboxes with the following options:

  * Flashcards
  * Rewriting notes
  * Practice tests
  * Group study
  * Watching videos
  * Other
* If "Other" is checked, display a text input box asking the user to describe their other study method
* At least one checkbox must be selected (required)
* The "Other" text box is required if "Other" is checked

---

## Form Behavior

* All fields are required; display inline validation error messages if the user submits without completing a field
* The "Other" study method text box appears dynamically only when "Other" is checked
* On successful submission, display a thank-you confirmation screen showing a summary of the user's answers
* The thank-you screen includes two buttons:

  * "Submit Another Response" — resets the form
  * "View Results" — navigates to the results page
* The submit button shows "Submitting..." while the request is in flight
* If the database insert fails, display an error message and keep the form intact

---

## Results Page

A separate `/results` route displays aggregated, anonymized data from all submissions. Individual responses are never shown — only group-level summaries.

### Charts / Visualizations

1. **Total Responses** — a large number prominently displayed at the top
2. **Year in College** — vertical bar chart, one bar per year option, ordered chronologically
3. **Most Popular Study Methods** — horizontal bar chart, sorted by count descending. For respondents who selected "Other", use the text they entered in `other_study_method` as the label rather than displaying "Other". Normalize casing before counting (e.g. "Flashcards" and "flashcards" are treated as the same entry). These user-entered methods appear as individual bars alongside the standard options.
4. **Top Study Locations** — horizontal bar chart showing the most common study locations based on responses

Use a charting library (e.g. Recharts) for all visualizations.

### Navigation

* Home page has a short welcome message with a "Take the Survey" button linking to the survey form and a "View Results" button linking to the results page
* The results page has a "Home" button linking back to the home page
* The survey form has a "View Results" button in the header linking to the results page

## Database

I am using my own Supabase account. Run the following SQL in the Supabase SQL Editor to create the table and configure access.

### Create Table

```sql
create table study_survey_results (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz default now(),
  favorite_study_place text not null,
  grade_level text not null,
  study_hours text not null,
  study_methods text[] not null,
  other_study_method text
);
```

`study_methods` is stored as a PostgreSQL text array. `other_study_method` is nullable — only populated when "Other" is selected.

### Row-Level Security

Supabase enables RLS by default and will block all inserts and selects unless policies are configured. Because this is a public survey with no authentication, run the following:

```sql
alter table study_survey_results enable row level security;

create policy "Allow public insert"
  on study_survey_results for insert
  to anon
  with check (true);

create policy "Allow public select"
  on study_survey_results for select
  to anon
  using (true);
```

### Environment Variables

The app connects to Supabase using two environment variables:

| Variable                 | Description                                                 |
| ------------------------ | ----------------------------------------------------------- |
| `VITE_SUPABASE_URL`      | Your Supabase project URL (e.g. `https://xxxx.supabase.co`) |
| `VITE_SUPABASE_ANON_KEY` | Your Supabase anon/public key                               |

Both values are found in the Supabase dashboard under **Project Settings → API**.

* For local development, create a `.env` file in the project root with these values. Add `.env` to `.gitignore`.
* For production, add both as GitHub Actions secrets. The CI/CD workflow injects them at build time.

## Design & Quality Standards

### Appearance

* Simple, minimalist design throughout
* White or light neutral background with dark body text
* `#8A3BDB` is the primary accent color — use it for buttons, links, active/focus states, borders, and chart colors
* Do not use `#8A3BDB` for body text; it does not meet WCAG AA contrast requirements at small sizes. Use a dark neutral (e.g. `#1a1a1a`) for body text
* For any text rendered in the accent color (e.g. headings or labels), verify contrast passes WCAG AA before using it

### Responsive Layout

The app must be usable on mobile, tablet, and desktop screen sizes. Use a single-column layout on small screens. Do not assume desktop-only use — the survey will likely be completed on phones.

### HTML5 Validation

All pages must validate as HTML5 with no errors using the W3C Markup Validation Service.

### Accessibility — WCAG 2.1 Level AA

All pages must pass WCAG 2.1 Level AA. Key requirements:

* All form inputs must have a programmatically associated `<label>` element
* Inline validation error messages must be programmatically associated with their fields using `aria-describedby`
* The "Other" study method text input must be keyboard accessible when it appears dynamically — focus should move to it automatically when "Other" is checked
* All images and icons must have appropriate `alt` text
* Color alone must not be used to convey meaning (e.g. error states must include text, not just a red border)
* All interactive elements must be reachable and operable via keyboard
* Use sufficient color contrast for all text — minimum 4.5:1 for normal text, 3:1 for large text

## Deployment

This app is deployed as an **Azure Static Web App** using Vite + React.

Because the app uses client-side routing (`/` and `/results`), a `staticwebapp.config.json` file must be included in the `public/` directory so Azure serves `index.html` for all routes instead of returning 404s on direct navigation or page refresh.

```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/assets/*"]
  }
}
```

This file is automatically copied to `dist/` during the Vite build.
